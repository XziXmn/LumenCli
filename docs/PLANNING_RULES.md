# Planning Rules

## 目的

本文件定义 Lumen 规划文档的分层规则，避免重新回到“任何新想法都新开一份平级规划文档”的状态。

## 真源分层

### 1. `docs/ROADMAP.md`

适合写入：

- 项目定位
- 当前主线
- Stage 级阶段路线
- 近期焦点
- 中长期方向
- 明确不做与暂缓

不适合写入：

- 逐步施工步骤
- 具体测试命令清单
- 大量历史背景复述

### 2. `docs/CAPABILITY_MATRIX.md`

适合写入：

- 具体能力项
- 能力状态
- 来源、落点、约束、下一动作
- 已完成但仍需长期维护的能力

不适合写入：

- 详细实现方案
- 阶段叙事
- 与单一专题强绑定的设计细节

### 3. `docs/superpowers/plans/*.md`

适合写入：

- 当前活跃专题
- 可执行步骤
- 定向验证命令
- 文件级影响范围

只有在满足以下条件时才新开专题计划：

1. 主题足够独立
2. 已经进入近期焦点
3. 需要明确施工顺序与验证门禁
4. 不能只靠在 `ROADMAP` 或 `CAPABILITY_MATRIX` 中补几行来表达

### 4. `docs/archive/`

适合放：

- 历史 planning / spec / blueprint
- 已废弃路线
- 只保留追溯价值的设计资料

Archive 文档默认不是当前真源。  
如果某份 archive 文档仍有参考价值，应在：

- `ROADMAP.md`
- `CAPABILITY_MATRIX.md`
- 或新的专题计划

中被重新提炼，而不是直接恢复为主入口。

### 5. `docs/superpowers/specs/*.md`

适合放：

- 已完成 brainstorming 但尚未进入实现施工的设计说明
- 对规划体系本身的结构重构说明
- 需要先达成设计共识、再决定是否落到 roadmap 或专题计划的文档

不适合放：

- 当前主路线图
- 能力账本
- 正在执行的专题施工图

如果某份 spec 已经被采纳，其结论应继续下沉到：

1. `ROADMAP.md`
2. `CAPABILITY_MATRIX.md`
3. 或 `docs/superpowers/plans/*.md`

spec 本身保留为设计决策记录，而不是长期真源。

## 更新规则

### 新能力进入主线时

至少同步更新：

1. `CAPABILITY_MATRIX.md`
2. 如果影响阶段优先级或项目方向，再更新 `ROADMAP.md`
3. 如果需要施工图，再新增或更新 `docs/superpowers/plans/*.md`

### 能力完成时

至少同步更新：

1. `CAPABILITY_MATRIX.md` 状态
2. `ROADMAP.md` 的近期焦点或阶段说明（如果该能力原本是重点）
3. 相关专题计划顶部的状态说明

### 能力被放弃或降级时

必须同步更新：

1. `CAPABILITY_MATRIX.md`
2. `ROADMAP.md` 的“不做与暂缓”
3. 对应专题计划或旧文档的降级说明

## 中文注解规范

### `ROADMAP.md`

采用讲解型中文。每个 Stage 至少回答：

1. 在解决什么问题
2. 为什么现在做
3. 包含什么
4. 不包含什么
5. 完成后会带来什么变化

### `CAPABILITY_MATRIX.md`

采用矩阵 + 简洁中文注解。每项至少说明：

1. 这项能力做什么
2. 当前做到什么程度
3. 为什么还要保留或推进
4. 下一步最合理的动作是什么

### 专题计划

在文档开头补中文导读，说明：

1. 属于哪个 Stage
2. 解决总路线图中的哪一块
3. 与哪些已完成能力相关
4. 执行时不能破坏哪些硬约束

## 兼容性约束

涉及以下内容时，文档必须显式说明兼容边界：

1. `.lumen/` 迁移来源
2. 上游 Pi 同步策略
3. 扩展 API 与 core UI 所有权
4. Windows / PowerShell 工作流

尤其是 `.lumen/` 迁移：

- 只能写成“旧资产迁移来源”
- 不能写成运行时 fallback 读取面
- 若未来要缩减迁移范围，必须单独立项并写清迁移策略

## 禁止事项

1. 不要重新使用旧 `Phase 3A/3B/3C` 编号作为当前主线
2. 不要让专题计划承担项目总路线图职责
3. 不要在 archive 文档中继续追加当前主线内容
4. 不要为了文档整洁把迁移来源误写成运行时兼容读取目标
